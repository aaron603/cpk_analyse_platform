💡 核心诉求Core Requirements

在"所有数据（含失败）"模式下，不仅要分析 CPK 统计数据，更要：
In "all data (incl. failures)" mode, beyond CPK statistics we also need to:

  • 自动发现所有失败条码（含测试不完整的情况）Auto-detect all failed barcodes (incl. incomplete tests)

🎯 终极目标Ultimate Goal

构建一个自我成长的故障知识库：
Build a self-growing fault knowledge base:

  • 每次分析 → 自动提取故障模式 → 更新规则库Each analysis → auto-extract fault patterns → update rules

  • 同一条码跨站/跨次数据 → 暴露环境差异、工装差异Same barcode cross-station data → exposes fixture / environment differences
  • 历史数据 + 新数据 = 越来越精准的故障分类能力Historical + new data = increasingly accurate fault classification
本质上是一套轻量级FDC（故障检测与分类）系统，聚焦 RRU/PA 产线场景。Essentially a lightweight FDC (Fault Detection & Classification) system focused on RRU/PA production lines.

diff 算法Diff Algorithm
def _diff_pass_fail(pass_items, fail_items):
    """
    pass_items: {item_name: value}
    fail_items: {item_name: value}
    返回 / Returns: [{
      item, fail_val, pass_val,
      lsl, usl,
      delta,         # fail_val - pass_val
      direction,     # 'degraded_high'/'degraded_low'/'na'
      severity       # 'critical'(超限/over-limit)/'warn'(>50%范围)/'ok'
    }]
    """

优先级 / Priority：仅输出有差异（delta 不为 0）的项，按 |delta| 降序排列，最多保留 Top 20。
Output only items with delta ≠ 0, sorted by |delta| descending, top 20 max.

场景A：A台 Fail → B台 Pass

极有价值的数据对：A/B 台测量值差异 = 环境差/工装差/仪器差的直接体现。高度提示测试设备问题而非 DUT 硬件问题。Highly valuable data pair: A/B measurement diff = direct evidence of environment/fixture/instrument differences. Strongly suggests test equipment issue, not DUT hardware.

七、业界最佳实践参考Ⅶ. Industry Best Practice References

🏭 FDC — 故障检测与分类（Fault Detection and Classification）

半导体制造领域标准方法（SEMI E58）。核心思想：在测试过程中实时检测偏差，自动分类并触发相应处置动作。
Standard method in semiconductor manufacturing (SEMI E58). Core idea: real-time deviation detection during testing, auto-classify and trigger corresponding actions.

常见分类维度：
Common classification dimensions:
设备工艺（Chamber/Tool）→ 配方（Recipe）→ 产品（Lot/Wafer）→ 部件（Die）
Tool → Recipe → Lot/Wafer → Die

📌 在本平台的映射：工站机台 → 产品大类 → 条码 → 测试项，与 FDC 四层分类高度一致。Platform mapping: Station → Product Type → Barcode → Test Item — aligns well with FDC's four-tier classification.

📊 SPC — 统计过程控制（Statistical Process Control）

用控制图（Shewhart Chart）监控关键参数的时间序列趋势。Western Electric Rules 定义了 8 种失控判据，如连续 8 点在均值同侧 = 过程偏移信号。
Uses control charts (Shewhart) to monitor key parameter trends. Western Electric Rules define 8 out-of-control criteria, e.g. 8 consecutive points on same side of mean = process shift signal.

在测试数据分析中：对每个测试项的历史数据画 Xbar-R 图，早期发现系统性漂移（如仪器老化、温漂）。
In test data analysis: plot Xbar-R charts per test item to detect systematic drift early (instrument aging, temperature drift).

📌 env_comp/*.csv 校准漂移检测是 SPC 的直接应用场景（平台待实现功能）。env_comp/*.csv calibration drift detection is a direct SPC application (planned platform feature).

🔬 Pass/Fail 差异分析（Marginality Analysis）

识别"边缘通过"（Marginal Pass）的 DUT：虽然 pass 了，但多个指标已接近限值，下次测试或老化后可能失败。业界常用方法：
Identifies "marginal pass" DUTs: passed but multiple metrics near limits — may fail on retest or after aging. Common industry methods:

  • Guard Band：比规格限更严的内部门限 / tighter internal threshold than spec limit
  • Margin Score：(value - limit_center) / (limit_range/2)，得分接近 ±1 表示边缘 / score near ±1 = marginal
  • Correlation Plot：两个强相关项同时劣化 = 共同原因故障 / two correlated items degrade together = common-cause fault
📌 diff_items 中的 severity 字段（critical/warn/ok）即是简化版 Margin Score。The severity field (critical/warn/ok) in diff_items is a simplified Margin Score.

🔄 持续学习规则库（Knowledge-Based Diagnosis）

航空发动机、工业机器人领域常用"专家系统 + 机器学习"混合架构：
Aero-engine and industrial robotics commonly use a expert system + machine learning hybrid:

  • 结构化规则（专家编写）+ 历史案例（数据驱动）联合打分 / structured rules (expert) + historical cases (data-driven) joint scoring
  • 新案例积累后，自动推荐新规则候选（如本方案的 rule_suggestions） / new cases auto-suggest new rule candidates (like our rule_suggestions)
  • 人工确认后规则入库，下次可用 → 形成人机协作的知识进化回路 / human-confirmed rules enter DB → forms a human-AI collaborative knowledge evolution loop
📌 本方案的 YAML 知识输入 + DB 持久化 + Ollama LLM 组合，正是轻量版此架构的落地实现。Our YAML input + DB persistence + Ollama LLM combination is a lightweight implementation of this architecture.

📍 Defect Pareto + Root Cause Matrix

可靠性分析的标准工具组合：Standard reliability analysis tool combination:

  • Pareto 图：80% 的失败来自 20% 的根因，优先解决高频项 / 80% of failures from 20% of root causes — fix high-frequency items first
  • 关联矩阵：哪些测试项经常同时失败（共因故障特征） / which test items frequently fail together (common-cause signature)
  • 时序图：某类故障在哪个时间段集中出现（批次问题 / 设备维修后） / when does a fault type cluster (batch issue / post-maintenance)
📌 _generate_fail_patterns() 的 top_failed_items 即 Pareto 的数据基础；共失败关联矩阵是 P2 待实现功能。top_failed_items in _generate_fail_patterns() is the Pareto data foundation; co-failure correlation matrix is a P2 planned feature.

🤖 LLM 辅助诊断（增强版） / LLM-Assisted Diagnosis (Enhanced)

已有 Ollama 集成（qwen2.5:7b）。业界经验：Ollama integration (qwen2.5:7b) already exists. Industry experience:

  • 提示词工程：结构化上下文 > 原始日志 dump；限制字数；要求 JSON 输出 / structured context > raw log dump; limit token count; require JSON output
  • RAG 增强：将历史已分类案例作为 few-shot 示例注入提示词 / inject historical classified cases as few-shot examples
  • 置信度阈值：LLM confidence < 0.6 时，保留"未分类"等人工审查 / when LLM confidence < 0.6, keep as "unclassified" for human review
📌 diff 分析结果可作为额外上下文注入 Ollama 提示词，提升诊断精准度。Diff analysis results can be injected into Ollama prompts as additional context to improve diagnostic accuracy.

九、关键技术决策Ⅸ. Key Technical Decisions

决策 1：full measurements 存在哪里？Decision 1: Where to store full measurements?

方案A 存入 fault_records.all_measurements JSON 字段（新增列，迁移兼容）
Store in fault_records.all_measurements JSON column (new column, migration-compatible)

方案B 只在内存中做 diff，不持久化原始全量数据
Diff in memory only, do not persist raw measurement data

推荐 A：持久化后可随时重新分析，不需要重新读取原始文件。字段可选填，不影响现有记录。Recommend A: persisted data can be re-analyzed anytime without re-reading raw files. Nullable column — does not affect existing records.
决策 2：无 pass 记录的失败条码如何处理？Decision 2: Handling failed barcodes with no pass record?

部分条码只有失败记录，从未测试通过。此时无法做 diff。
Some barcodes only have fail records — never passed. Diff is impossible in this case.

处理方式

  • barcode_comparisons.pass_record_id = NULL
  • diff_summary = "无对应 pass 记录 / No matching pass record"
  • 在报告中标注"首次测试即失败"，此类 DUT 优先怀疑硬件问题Mark as "failed on first test" in report — suspect hardware issue first
决策 3：YAML 中的规则冲突如何处理？Decision 3: Handling YAML rule conflicts?

当新 YAML 的 keywords 与现有规则高度重叠时：
When new YAML keywords highly overlap with existing rules:

  • 精确匹配（同 keywords + 同 fault_type）→ 更新 suggestion（版本升级）Exact match (same keywords + same fault_type) → update suggestion (version upgrade)
  • 同 keywords 但不同 fault_type → 警告用户，让用户选择"覆盖/跳过/新增"Same keywords, different fault_type → warn user, let user choose overwrite/skip/add
  • 全新 keywords → 直接新增All-new keywords → insert directly
决策 4：all_with_fail 与现有 all 模式的关系？Decision 4: Relation between all_with_fail and existing 'all' mode?

建议：all_with_fail 作为 all 模式的超集升级版：
Recommendation: all_with_fail as a superset upgrade of all mode:

  • 在 run_fault_analysis(mode=) 参数中新增 'all_with_fail'Add 'all_with_fail' to the mode parameter of run_fault_analysis()
  • 比 'all' 多做：①pass 记录提取全量测量值；② diff 分析；③跨站比对Does more than 'all': ① extract full pass measurements; ② diff analysis; ③ cross-station comparison
  • 比 'fail_only' 多做：保留 pass 记录作为参照基准Does more than 'fail_only': retains pass records as reference baseline
